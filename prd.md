# Living Almanac — Product Requirements Document (PRD)

## 1) Product Summary

Living Almanac is an AI-native, mobile-first knowledge-scouting platform where young people capture elder/community knowledge, turn it into structured cultural assets, publish selected knowledge to a beautiful world map, and monetize verified entries through licensing.

Core promise:

* For youth: earn, learn, gain identity, and build status.
* For elders/community: preserve knowledge, receive attribution and royalties, and gain dignity.
* For buyers/institutions: access provenance-verified, licensable cultural knowledge.

## 2) Product Vision

Build the most trusted platform for community-owned living knowledge in Southeast Asia.

## 3) Product Principles

1. Community ownership over extraction.
2. Capture must feel like play, not paperwork.
3. Verification must build trust without killing momentum.
4. Public surfaces must be beautiful enough to share.
5. Every meaningful action should reinforce identity: “I am a Knowledge Scout.”
6. Royalty and attribution must be radically transparent.

## 4) User Segments

### Primary

* Youth Scouts (15–25): rural/peri-urban, mobile-native, status-driven, income-seeking, creator-like behaviors.
* Elders / Knowledge Holders: domain experts in crafts, agriculture, recipes, rituals, folklore, weather reading, etc.

### Secondary

* Coordinators / Verifiers: commune officers, NGOs, teachers, moderators.
* Buyers: brands, studios, publishers, tourism operators, researchers.
* Institutions: schools, ministries, NGOs.
* Public audience: map explorers, supporters, tourists.

## 8) User Flows

### Flow 1: First-time Scout activation

1. Install app
2. Choose language and community/region
3. Watch/scroll short “become a scout” narrative
4. Accept mission + ethics/consent basics
5. Create profile
6. Receive first quest: “Document one family technique/story/recipe”

### Flow 2: Capture a knowledge entry

1. Tap “Start Mission”
2. Select category
3. AI suggests interview prompts
4. Input text / capture photo / capture short video
5. AI structures and enhances content
6. User reviews summary and fills gaps
7. Add elder attribution + consent
8. Submit for verification
9. Receive XP and pending status

### Flow 3: Revision and verification

1. Verifier reviews entry
2. Entry marked approved / revise / rejected
3. If revise: Scout gets clear fix requests
4. Scout resubmits
5. Approved entry appears on profile and map
6. Scout gets badge / XP boost

### Flow 4: Public discovery and sharing

1. Approved entry populates world map
2. Scout receives share card
3. Shares to TikTok / Instagram / Zalo / Threads
4. Friends open map / profile / app landing experience
5. New users onboard via shared content

### Flow 5: Licensing / earnings

1. Buyer discovers entry or ops-curated package
2. Buyer requests rights package
3. Internal review + approval
4. Contract/payment completed
5. Royalty split posted to ledger
6. Scout + elder notified
7. Scout shares “I got paid for this” moment

## 13) Tech / Component Recommendations

### Prototype stack

* Frontend web app: Next.js + TypeScript
* Mobile-first responsive UI first; later React Native / Expo if needed
* Styling: Tailwind CSS
* Components: shadcn/ui
* Icons: lucide-react
* Forms: react-hook-form + zod
* Data fetching/state: TanStack Query
* Maps: Mapbox GL or react-map-gl
* Motion: Framer Motion
* Text capture and processing: AI structuring and translation mocked in prototype
* Auth: Clerk or Supabase Auth
* Backend prototype: Supabase (Postgres, Storage, Row Level Security)
* Search: Postgres FTS for MVP, vector later if needed
* Charts/admin analytics: Recharts

* Tokenized spacing, typography, color, radius, shadows
* Badge/rarity component family
* Map card component family
* Submission state patterns
